Published 2019-05-25

Mysql总结

一、 Mysql事务

1.1 mysql事务的定义

  • 事务是数据库并发控制的基本单位
  • 食物可以看作是一系列SQL语句的集合
  • 事务要么全部执行成功，要么全部执行失败（回滚）

1.2 事务的四个基本特性（ACID）

  • 原子性（Atomicity）: 一个事务中所有操作全部完成或失败
  • 一致性（Consistency）: 事务开始和结束之后数据完整性没有被破坏
  • 隔离性（Isolation）:允许多个事务同时对数据库修改和读写
  • 持久性（Durability）: 事务结束后，修改是永久的不会丢失

1.3 事务隔离级别

  1. 读未提交： 别的事务可以读取到未提交的改变
  2. 读已提交： 只能读取到已经提交的数据
  3. 可重复读： 同一个事务先后查询结果一样（mysql默认隔离级别）
  4. 串行化： 事务完全串行化执行，隔离级别最高，执行效率最低

二、 乐观锁和悲观锁

  • 悲观锁： 现获取锁再进行更新操作。一锁二查三更新，可通过SQL select for update实现。
  • select for update是行锁还是表锁只取决于能否使用索引，能则为行锁，否则为表锁（比如like、<>操作作会导致索引失效）。
  • 乐观锁： 先假设我修改的时候别人不会去修改，修改完毕后发现数据已经变了就回滚（一般通过版本号或时间戳结合业务代码实现）
  • 根据响应速度、冲突频率、重试代价来判断使用哪种锁

三、 Mysql索引原理及优化

3.1 索引简介

  • 索引是数据表中一个或者多个列进行排序的数据结构
  • 索引能大幅提升检索速度
  • 创建、更新索引本身也会耗费时间和空间（并不是索引越多越好）

3.2 B-树

3.2.1 查找结构进化史
  1. 线性查找： 一个个找、实现简单、太慢
  2. 二分查找：有序、简单、要求是有序的、插入特别慢
  3. hash： 查询快、占用空间、不适合存储大规模数据
  4. 二叉查找树： 插入和查询很快、无法存大规模数据、复杂度退化
  5. 平衡树： 解决二叉树退化问题；节点非常多的时候，依然树很高。
  6. 多路查找树： 一个父亲多个孩子节点，节点过多，树不会特别深
  7. 多路平衡查找树： B-树， 无法进行范围查找
  8. B+树

3.3 B+树

  • B+树是B-树的变形
  • 只有叶子节点上带有指向数据的指针（非叶子节点可以节约空间、增加树的）
  • 增加了树的度，即一个节点可以有更多个孩子，那么整体的树高会降低，我们知道树的查询复杂度和树高是相关的
  • 叶子节点通过指针相连，实现了范围查询

3.4 索引最佳实践

3.4.1 什么时候创建索引
  • 经常用作查询条件的字段（where条件）
  • 经常用作表连接的字段
  • 经常出现在order by，group by之后的字段
3.4.2 需要注意的点
  • 非空字段(not null), mysql无法对空值作运算、比较。添加默认值。
  • 区分度高，离散度大，作为索引的字段尽量不要有大量相同的值
  • 索引的长度不要太长（比较耗费时间）
3.4.3 索引什么时候失效
  • 模糊匹配
  • 类型隐转（动态语言中需要注意）
  • 没有满足最左前缀原则
  • 所有索引失效的问题都是由于B+树的key没有办法比较
3.4.4 聚集、非聚集索引
  • 聚集还是非聚集指的是B+树叶子节点存的是指针还是数据记录
  • MyIsam索引和数据分离，使用的是非聚集索引（对应数据目录下一个表会有两个文件.MYD和.MYI）
  • Innodb数据文件就是索引文件，主键索引就是聚集索引
  • innodb辅助索引中实际存储的是主键的值，通过找到主键进一步找到目标数据

python字符编码

如果不懂字符编码，将会是程序员一生之痛。

​

​

一、python3的字符编码

首先明确下面几点:

  1. python3内存中都是解码过的unicode(万国码)二进制
  2. 硬盘中存的是经过编码的二进制(gbk,ascci等)
  3. 我们看到的字符输出等都是经过应用程序从内存中编码得到的
  4. 控制python以什么编码方式读取代码到内存: #coding: utf-8(gbk)
  5. unicode的二进制在内存中怎么都不会乱码

​

二、编码与解码

由上图可见，乱码的情况只可能发生在:

  1. 读取字符到内存时的解码方式和存取文件时的编码方式不同

  2. 读取字符到内存时的解码方式和输出时的编码方式不同

    ​

    ​

三、python2的字符编码

python2由于历史遗留问题默认使用的是ascci编码方式，因为1989年python诞生，而unicode是在1994年才正式发布。

python2与python字符编码的最大区别:

  • 默认使用ascci编码到内存

这样就导致了python2内存中无法做到怎么都不会乱码，需要先解码，再编码。

​

​

四、总结

  1. 以什么方式读的，就以什么方式存或者取，可以保证字符不会乱码。
  2. 更加坚定的使用python3。
